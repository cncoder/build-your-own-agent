Ch 02 · 问题引入

为什么单次 API 调用不够？

场景：让 AI 帮你订机票

✗ 单次 API 调用
用户：帮我订明天北京→上海的机票
AI：您可以在携程、飞猪或官网购票。建议选择早班机……
← 只能给建议，无法真正执行
VS
✓ ReAct 循环
Thought: 需要先搜索明日航班
Action: search_flights("北京", "上海", "明日")
Observation: 找到 12 个航班，最早 07:00 ¥892
← 每步都在真实执行
单次调用缺少「执行→观察→再推理」的回路——这个回路，就是 ReAct 要解决的核心问题。

Ch 02 · ReAct 论文

Yao et al., 2022 arxiv: 2210.03629

ReAct: Synergizing Reasoning and Acting in Language Models
Shunyu Yao · Jeffrey Zhao · Dian Yu · Nan Du · Izhak Shafran · Karthik Narasimhan · Yuan Cao
Princeton University · Google Brain · 2022-10-06
核心发现：把 推理（Reasoning） 和 行动（Acting） 交织在同一个调用流程里，比纯推理或纯行动的效果显著更好。
🧠

纯 Chain-of-Thought（CoT）

LLM 内部推理，但无法获取外部真实信息，容易产生幻觉。推理是在「假设世界」里进行的。

⚡

ReAct = Re + Act

推理和行动交替进行。每次 Observation 都是真实的外部反馈，下一步推理基于真实世界状态。

Reasoning + Acting = Agent 的原子单元

Ch 02 · 状态机
LLM 推理层 工具执行层 真实反馈层

ReAct 状态机

每个节点的 LLM 输入 / 输出

Thought · 推理

LLM 接收完整 messages + 工具定义列表

输出： 文本推理 + type:"tool_use" block

Action · 执行

工具代码（Python/TS）实际运行

输出： 真实执行结果（不是 LLM 猜的）

Observation · 观察

工具结果写入 messages 数组（role: "user", type: "tool_result"）

输出： 账本追加 → 下一轮 Thought 的输入

Anthropic 五大工作流模式

Building Effective Agents · 2024-12-19
01 Augmented LLM LLM + 检索 / 工具 / 记忆的基础能力增强 单步问答增强

Ch 02 · Workflow vs Agent

Anthropic 铁律：先找最简方案

"
Start with the simplest solution. When in doubt, don't use agents. If you need agents, use the simplest architecture that works. Only add complexity when it demonstrably improves outcomes.
— Anthropic · Building Effective Agents · 2024-12-19

用 Prompt Chaining 当

步骤固定可预测：流水线式任务，每步顺序不变，不需要动态决策。
成本最低，最易调试。

用 Orchestrator-Workers 当

任务需要自主拆解：步骤在执行前无法完全确定，需要 LLM 动态规划。

不用 Agent 当

结果可确定性高：一次调用能搞定的不要加循环。Agent 带来复杂性和不确定性。

构建跨厂商 agent 必须了解的格式差异

维度 Anthropic OpenAI

⚠ 参数格式 input: { } 直接对象 arguments: "{ }" 字符串需 parse
结果角色 role: "user" + type: "tool_result" role: "tool"

最易踩坑： OpenAI 的 function.arguments 是字符串，需要 JSON.parse() 才能用。 Anthropic 的 input 直接是对象。Provider 抽象层负责抹平这个差异。

// claude-code-source/source/src/query.ts
async function* queryLoop(params, consumedCommandUuids) {
while (true) { // ReAct 循环主体
// ... microcompact, autocompact, blocking_limit
// ① Thought: 调用模型
for await (const message of callModel(...)) {
if (tool_use) streamingToolExecutor.addTool(block) // ② Action
}
if (stop_reason == 'end_turn' && !tool_use) {
handleStopHooks()
return { reason: 'completed' } // 出口
}
// ③ Observation: 收集工具结果
for await (const update of streamingToolExecutor.getRemainingResults()) {
yield update.message // tool_result 消息
}
state = { messages: [...messages, ...assistant, ...tools] }
} // while(true)
}

代码 ↔ 理论对照

每行都对应 ReAct 状态机的一个节点

Thought · callModel()

SSE 流式调用，接收完整 messages 上下文，输出 text + tool_use blocks

Action · streamingToolExecutor

流式传输中即开始并发执行工具，并发上限 10（toolOrchestration.ts:10）

Observation · getRemainingResults()

tool_result 消息 yield 给上层，追加到 messages → 下一轮 Thought 的输入

Exit · end_turn + no tools

stop_reason == 'end_turn' 且无工具调用时退出循环，handleStopHooks() 执行清理

Ch 02 · Messages 账本

messages 数组是 Agent 的完整账本

示例任务：「帮我统计项目有多少行 TypeScript 代码」

初始状态
role: "user"
content: "帮我统计…"
→
Turn 1 · Thought+Action
role: "user"
content: "帮我统计…"
NEW
role: "assistant"
content: [{ type: "text", text: "我来用 shell…" }, { type: "tool_use", id: "tu_1", name: "shell", input: { cmd: "find…" } }]
→
Turn 1 · Observation
role: "user"
role: "assistant" + tool_use
NEW
role: "user"
content: [{ type: "tool_result", tool_use_id: "tu_1", content: "3847 total" }]
→
Turn 2 · 最终回复
role: "user"
role: "assistant" + tool_use
role: "user" + tool_result
NEW
role: "assistant"
content: "项目共有 3847 行 TS 代码。"
每轮都在往账本里追加真实条目。信息密度随轮数线性增加，但模型每轮看到完整历史——这就是 ReAct 比单次调用强的数学原因。

Ch 02 · Design Note
⚡ DESIGN NOTE

Plan-and-Execute vs ReAct

为什么主流 agent 框架都选了 ReAct？

✓ ReAct

  • ✓ 每步基于真实 Observation 推理
  • ✓ 动态自适应：中途发现新情况可改变方向
  • ✓ 工具失败 → 下一步 Thought 处理错误恢复
  • ✗ LLM 调用次数多 → 成本较高

Plan-and-Execute

  • ✓ 步骤固定可审批：先让人 review 计划
  • ✓ 步骤完全确定时成本更低（少 LLM 调用）
  • ✗ 计划是对未来世界的假设，执行时发现信息不对，但计划已固定
  • ✗ 难以中途等待人类审批，难以处理意外
结论： ReAct 适合探索性、动态的任务（多数真实场景）；Plan-and-Execute 适合结构化、可预测的任务。高级架构（如 LangGraph）可把两者结合：粗粒度 Plan + 细粒度 ReAct 执行。

Ch 02 · 本章小结

三条金句

01

ReAct = 每步都基于真实 Observation

把推理和行动交织，不是先想清楚再执行。每次 Observation 都是真实的外部反馈，而非 LLM 的内部幻想。

02

关键约束放工具代码层，不放 Prompt 层

hermesbot 教训：AI 会忽略 Prompt 里的规则，但无法绕过代码级检查。风控、验证、业务规则——放在 Observation 的生产者（工具代码）里。

03

Start Simple：80% 场景 Prompt Chaining 就够了

Anthropic 铁律：只在有明确证据表明能提升效果时才加复杂度。Provider 抽象层统一格式差异，上层 agent 代码不感知厂商。
