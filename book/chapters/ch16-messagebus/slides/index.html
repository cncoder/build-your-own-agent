问题：为什么直连会出问题？

一个 Channel 崩了，Lena 全挂

❌ 紧耦合（lena-v1.1）

Telegram ──▶ AgentLoop
Discord  ──▶ AgentLoop
WebSocket──▶ AgentLoop
HTTP     ──▶ AgentLoop

// N 个 channel × M 个 handler
// = N×M 条耦合线
// Telegram 超时 → AgentLoop 崩溃
// Discord 消息也不响应了

✅ MessageBus 解耦（lena-v1.2）

Telegram ──┐         ┌──▶ AgentLoop
Discord  ──┤         ├──▶ Logger
WebSocket──┤──▶ Bus ─┤
HTTP     ──┘         └──▶ Notification

// N + M，不是 N×M
// Telegram 超时 → 只影响 Telegram
// 其他 channel 照常运行

三个核心痛点：① 一个 channel 崩了影响所有 ② 加 Subscriber 要改所有 Publisher ③ 不能热插拔

class MessageBus extends EventEmitter {
  // channel → handlers 映射
  private handlers: Map<string, Set<MessageHandler>>;
  // 全局 handlers（所有消息都触发）
  private globalHandlers: Set<MessageHandler>;
}

subscribe/unsubscribe（line 25-61）

subscribe(channelType, handler) {
  this.handlers.get(channelType).add(handler);
  // Set.add：幂等，不重复注册
}
unsubscribe(channelType, handler) {
  handlers.delete(handler);
  if (size === 0) this.handlers.delete(channelType);
  // 清理空 Set，防内存泄漏
}

publish（line 66-87）

async publish(message) {
  // channel-specific handlers 并发执行
  const promises = handlers.map(h =>
    this.safeHandlerCall(h, message));
  await Promise.all(promises);

  // global handlers 并发执行
  const gp = globalHandlers.map(h =>
    this.safeHandlerCall(h, message));
  await Promise.all(gp);
}

Promise.all = 并发执行，AgentLoop 和 Logger 同时收到消息，互不等待

Safety 支柱：safeHandlerCall 错误隔离

8 行代码，实现隔仓设计

nano-claw bus/index.ts:92-99

private async safeHandlerCall(
  handler: MessageHandler,
  message: ChannelMessage
): Promise<void> {
  try {
    await handler(message);
  } catch (error) {
    logger.error('Error in handler', error);
    this.emit('error', { error, message });
    // 注意：没有 rethrow！
  }
}

演示：broken_handler 不影响其他

# 加入一个故意抛错的 handler
bus.subscribe_all(broken_handler)

# 发布消息
await bus.publish(msg)

# 输出：
# [AgentLoop] 正常处理 ✓
# [Logger]    正常记录 ✓
# [BrokenHandler] 抛错 ✗ ← 隔离！
# 其他 handler 不受影响

Bulkhead Pattern（隔仓设计）：舰船隔仓保证一仓进水不沉船；safeHandlerCall 保证一个 handler 崩溃不停机

globalHandlers vs channel-specific handlers

按关注点分类，不是按喜好

类型 注册方式 触发条件 典型用途
channel-specific bus.subscribe("telegram", h) 只在 telegram 消息到来时 AgentLoop（区分来源）
global bus.subscribe_all(h) 任何 channel 的消息 Logger / Analytics / 审计

判断规则：业务逻辑 → channel-specific；横切关注点（日志、监控、审计）→ global

真实案例（案例 12.1）

main agent（Discord）直接给 mengqian（飞书）发消息 → 报 Cross-context messaging denied

正确做法：通过 cron systemEvent 在 mengqian 自己的 session 触发，或加入 subagent allowlist

MessageBus 的 channel-specific 设计避免了这种跨 context 的直接调用

Design Note：什么时候引入 MessageBus？

Bus vs 直接调用——不是永远用 Bus

场景 推荐方案 原因
1 个 channel，1 个处理逻辑 直接调用 Bus 是不必要的间接层
2-3 个 channel，固定逻辑 直接调用 + try/except 简单场景简单处理
多 channel × 多 Subscriber MessageBus O(N×M) → O(N+M)
需要热插拔 MessageBus subscribe/unsubscribe 天然支持
需要错误隔离 MessageBus safeHandlerCall 提供保障

AutoGen 类比：Microsoft AutoGen Core 层的 event-driven runtime，agent 之间通过消息传递而非直接调用。nano-claw MessageBus 是 134 行的极简版，原理完全一致。

本章小结

六个关键洞察

1. N×M → N+M

MessageBus 把耦合复杂度从 N×M 降到 N+M

2. safeHandlerCall

8 行代码，隔仓设计，单 handler 崩溃不停机

3. globalHandlers vs specific

横切关注点用 global；业务逻辑用 channel-specific

4. 热插拔

subscribe/unsubscribe，不重启 Lena，动态增减 channel

5. 幂等性铁律

事件触发必须检查是否已有实例，否则双进程 bug（案例 12.2）
