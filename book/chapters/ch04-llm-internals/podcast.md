【窦文涛】你打开 Anthropic 文档，看到 Claude Opus 4、Sonnet 4.5、Haiku 3.5 并排排列，直觉选了 Opus，"更聪明嘛"。三周后账单来了：847 美元。同样任务用 Haiku 只需要 23 美元。质量测试显示 Haiku 工具调用正确率 91%，Opus 93%。你为 2% 质量多花了 37 倍的钱。这不是特例，是没有正确直觉框架的必然结果。Anthropic 在 Building Effective Agents 里说：选型要平衡能力、速度、成本——就像从工具箱选工具，不会用大锤挂画框。但你得理解模型到底差在哪里。本章八个心智模型，完整覆盖这个前提。

【周迅】先从最基础的讲？

【窦文涛】对，第一个心智模型：串行生成，也就是为什么 LLM 输出是一个 token 一个 token 地往外吐。每生成一个 token，模型都要把前面所有内容重新"看"一遍，然后决定下一个。这叫自回归生成。RNN 时代的问题是信息会衰减——处理第 1000 个 token 时，第 1 个 token 的信息已经稀释得差不多了，像电话游戏。2017 年 Vaswani 等人发表的 Attention Is All You Need 彻底改变了这件事：每个 token 都能直接"看到"序列中任意其他 token，距离不造成遗忘。

【窦文涛】串行生成的工程后果：生成第 100 个 token 时必须已经生成了前 99 个，无法并行输出。写报告、生成代码这类长回复任务天生比短回复慢。设计 agent 时要给足 timeout，或者把任务拆成多个短回复步骤——这是架构决策，不是 prompt 优化。

【周迅】那平方复杂度是什么意思？

【窦文涛】第二个心智模型：O(n²)。Attention 的核心操作是序列中每个 token 要和其他所有 token 两两比较——n 个 token 就有 n×n 对比较。GPT-3 是 96 层 Attention，每层处理完整序列，1750 亿参数。上下文从 4K 增加到 128K，差了 32 倍，但计算量增加约 1024 倍（32 的平方）。处理一份 150K token 的 PDF，成本是处理 8K 上下文的约 18 倍，不是线性的——平方叠上去的。现代优化如 Flash Attention 能压低实际计算量，但平方关系是理解"长 context 为什么贵"的正确心智模型。

【窦文涛】一个实际问题叫注意力稀释：context 太长时，早期的关键指令容易被后续大量内容"淹没"，模型有时会忽略它。这是除了成本以外"不要无脑塞满 200K context"的第二个理由。

【周迅】所以 KV Cache 是为了解决这个计算开销的？

【窦文涛】部分是。第三个心智模型：KV Cache 和 Prompt Caching。Attention 计算时每个 token 生成三样东西：Query（我要问什么）、Key（我能回答什么）、Value（我的信息内容）。多轮对话时，前面轮次的 K 和 V 不会变——你不会修改历史消息。KV Cache 就是把已经算过的 K/V 缓存起来，下一轮只算新增 token，不重新算全部。

【窦文涛】数字：200 轮对话，每轮新增 50 tokens，不用 KV Cache 要重算全部约 10000 tokens；用了缓存每轮只算新增的 50 tokens，折算成 API 费用约省 85%。Prompt Caching 是 KV Cache 在 API 层的表现。Anthropic 用 cache\_control 标记可缓存的 prompt 前缀，命中后输入 token 价格降至 10%；OpenAI 对超过 1024 tokens 的 prompt 自动启用，命中后降至 50%。

【窦文涛】工程实践核心：把 system prompt、工具 schema、长文档放在消息开头，每轮变化的用户输入放在最后。这让缓存命中率最大化。最常见的坑是时间戳或当前状态放在 prompt 开头——每秒都在变，前缀永远匹配不上，缓存命中率归零。

【周迅】那 context window 为什么有上限，是硬件限制？

【窦文涛】对，第四个心智模型：Context Window 的限制。KV Cache 要住在 GPU 显存里，因为每次生成新 token 都需要访问它。70B FP16 模型光权重就需要约 140GB——单张 A100 80GB 装不下，必须多卡并行。支持 128K context 的 KV Cache 还需要再加约 64GB（每 1K tokens 占约 0.5GB），这就是为什么大 context 需要大量硬件。

【窦文涛】一个关键工程细节：超过 context window 的内容不会报错——API 会截断最早的内容，静默丢失。agent 在长会话中会突然"忘记"早期的关键指令，原因就在这里。这就是为什么 Claude 200K、Gemini 1M 是工程成就：他们找到了在可控成本下用 KV Cache 压缩和 Multi-Query Attention 等技术扩展 context 的方法。

【周迅】量化跟这个显存问题直接相关？

【窦文涛】对，第五个心智模型：量化。存储模型权重用多少比特，决定精度和显存占用。FP16 是 16 位浮点，70B 模型需要约 140GB 显存；INT4 是 4 位整数，70B 模型压到约 35-40GB，能跑在 4 张 RTX 4090 上——消费级 GPU 的入场券。BF16 是 Brain Float 16，指数范围和 FP32 一样宽，不容易溢出，A100 和 H100 对 BF16 有原生加速，是服务端训练的标准精度。

【窦文涛】量化的代价：INT4 质量损失在多数任务约 5-10%。数学推理和逻辑推理受影响大——精度损失在多步链中累积放大；分类、摘要、信息提取基本感知不到。好的量化实现如 GPTQ、AWQ、llama.cpp 的 Q4\_K\_M 质量影响很小。如果用 API，量化在服务端透明处理；Groq、Together 这类推理服务正是因为用了激进量化才能比官方 API 便宜很多。

【周迅】MoE 架构跟量化是不同维度的省成本方式？

【窦文涛】是的，第六个心智模型：Dense 与 MoE。传统 Dense 模型每次生成 token 时激活所有参数。Mixture of Experts 的思路是：把 FFN 前馈网络层分成多个专家，每次只路由给其中几个处理——如果有 64 个专家每次激活 2 个，实际计算量只有 1/32。但模型"见过的知识"来自所有专家训练——总参数大，激活参数小。DeepSeek-V3 总参数 671B，每 token 激活 37B，2025 年发布时质量接近 GPT-4o，推理成本约为其十分之一。

【窦文涛】关键认知：激活参数决定推理成本，不是总参数。Mixtral 8×7B 总 46B，但推理激活约 13B——性能接近 13B Dense 模型的计算量而不是 46B。当你看到便宜大参数模型，先查激活参数，再做选择。MoE 的代价是所有专家参数都要加载进显存，671B 意味着需要超过 1.3TB 显存存模型——但 API 服务用共享服务器摊平这个成本。

【周迅】那选型的实践框架是什么？

【窦文涛】第七个心智模型：选型框架。截至 2026 年初，agent 工具调用场景中，Claude Sonnet 系列在质量和成本比上持续保持第一梯队——Berkeley function calling leaderboard 2025 全年数据支持这个结论。几条有立场的铁律：工具调用首选 Claude Sonnet 系列，function calling 格式最稳定，返回格式遵从率高；中文任务优先 DeepSeek-V3 或 Qwen3，同样内容 Claude 消耗约 1.5 到 2 倍 token，且价格是 Claude 的十分之一以下；本地部署入门：显存 GB 除以 2 约等于 FP16 精度下能跑的参数量 B。

【窦文涛】成本估算在动手之前必须先做：日成本等于每次调用 tokens 乘以调用次数乘以 token 单价。每次调用 8K tokens，每天 1000 次，Claude Sonnet 4.5 输入 3 美元每百万、输出 15 美元每百万，日成本约 36 美元，月约 1080 美元。换 Haiku（输入 0.8、输出 4），月约 288 美元，节省 73%。质量测试先于成本决策——确认 Haiku 满足任务要求再降级。

【周迅】Reasoning 模型什么时候才值得用？

【窦文涛】第八个心智模型：Reasoning 模型的适用边界。Reasoning 模型在输出最终答案之前会生成大量"思考过程" token。DeepSeek 的推理模型处理一道多步推理数学题，内部思考消耗约 2000 到 8000 tokens，然后才输出答案——如果答案只有 50 tokens，thinking tokens 的成本可能是答案本身的 40 到 160 倍。

【窦文涛】关键限制：Reasoning 模型在 agent 工具调用里格式遵从率不稳定。普通模型训练成了"需要工具时输出这个 JSON 格式"，行为固定。Reasoning 模型在思考中可能改变主意，或者思考完后忘记按格式调用工具，甚至在思考里直接给出答案而不走工具——你的工具从来没被执行，loop 提前终止，用户拿到的是模型内部知识而不是工具实际返回的结果。

【窦文涛】用 Reasoning 模型的条件：任务是复杂推理（数学证明、多步逻辑、深度代码调试）、允许较高延迟、是"想清楚再做一件事"而不是"20 轮工具调用里每次做小决策"。默认用标准模型跑 agent loop，只在复杂推理节点切换到 Reasoning 模型——这是专用工具，不是通用升级。

【周迅】想深入学哪里找？

【窦文涛】Karpathy 的 zero-to-hero 系列把 124M 参数 GPT 压缩进 300 行代码，每步都有数字验证，是目前感受"从 bigram 到 Transformer"最直接的路径。Raschka 的 Build a Large Language Model From Scratch 在 Ch4 实现 GPTModel、Ch5 完成预训练，更系统。理解 LLM 内部不是使用 LLM API 的必要前提——但它能让你在做选型决策时知道自己在做什么，而不是凭感觉猜。下一章，构建"加工具不改核心"的注册机制，让 Lena 优雅扩展到 20 个工具。
