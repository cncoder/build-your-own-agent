【窦文涛】你打开 Anthropic 文档，看到 Claude Opus 4、Sonnet 4.5、Haiku 3.5 并排排列，直觉选了 Opus，"更聪明嘛"。三周后账单来了：847 美元。同样任务用 Haiku 只需要 23 美元。质量测试显示 Haiku 工具调用正确率 91%，Opus 93%。你为 2% 质量多花了 37 倍的钱。这不是特例，是没有正确直觉框架的必然结果。Anthropic 在 Building Effective Agents 里说：选型要平衡能力、速度、成本——就像从工具箱选工具，不会用大锤挂画框。但你得理解模型到底差在哪里。本章就从这个"你得先理解模型底层"的前提出发，把选型背后的机制讲清楚。

【周迅】那 token 这个最基本的单元先讲？

【窦文涛】对，第一个心智模型：串行生成，也就是为什么 LLM 输出是一个 token 一个 token 地往外吐。这叫自回归生成——每个 token 的输出依赖前面所有 token 的上下文，无法并行生成。RNN 时代的问题是信息会衰减——处理第 1000 个 token 时，第 1 个 token 的信息已经稀释得差不多了，像电话游戏。

【窦文涛】2017 年的 Attention Is All You Need 论文提出了 Transformer 架构。Transformer 的定义是：把 token 序列依次送进若干层 Attention 层，每层让每个 token 直接关注序列中其他所有 token，最后预测下一个 token 的概率分布。GPT-3、Claude、Gemini 都是 Transformer。它取代 RNN 的核心原因：每个 token 都能直接"看到"序列中任意其他 token，距离不造成遗忘。

【窦文涛】串行生成的工程后果：生成第 100 个 token 时必须已经生成了前 99 个，无法并行输出。写报告、生成代码这类长回复任务天生比短回复慢。设计 agent 时要给足 timeout，或者把任务拆成多个短回复步骤——这是架构决策，不是 prompt 优化。

【周迅】这个距离不造成遗忘听起来很好，那计算量呢？

【窦文涛】第二个心智模型：O(n²)。Attention 的核心操作是序列中每个 token 要和其他所有 token 两两比较——n 个 token 就有 n×n 对比较。GPT-3 是 96 层 Attention，每层处理完整序列，1750 亿参数。上下文从 4K 增加到 128K，差了 32 倍，但计算量增加约 1024 倍（32 的平方）。处理一份 150K token 的 PDF，计算量远不止是处理 8K 上下文的线性倍数——平方叠上去的，成本增幅会超过线性估算很多。现代优化如 Flash Attention 能压低实际计算量，但平方关系是理解"长 context 为什么贵"的正确心智模型。

【窦文涛】一个实际问题叫注意力稀释：context 太长时，早期的关键指令容易被后续大量内容"淹没"，模型有时会忽略它。这是除了成本以外"不要无脑塞满 200K context"的第二个理由。

【周迅】那每次都重算这么贵，有没有办法缓存？

【窦文涛】部分是。第三个心智模型：KV Cache 和 Prompt Caching。Attention 计算时每个 token 生成三样东西：Query（我要问什么）、Key（我能回答什么）、Value（我的信息内容）。多轮对话时，前面轮次的 K 和 V 不会变——你不会修改历史消息。KV Cache 就是把已经算过的 K/V 缓存起来，下一轮只算新增 token，不重新算全部。

【窦文涛】数字：200 轮对话，每轮新增 50 tokens，不用 KV Cache 要重算全部约 10000 tokens；用了缓存每轮只算新增的 50 tokens，折算成 API 费用约省 85%。Prompt Caching 是 KV Cache 在 API 层的表现。Anthropic 用 cache\_control 标记可缓存的 prompt 前缀，命中后输入 token 价格降至 10%；OpenAI 对超过 1024 tokens 的 prompt 自动启用，命中后降至 50%。

【窦文涛】工程实践核心：把 system prompt、工具 schema、长文档放在消息开头，每轮变化的用户输入放在最后。这让缓存命中率最大化。最常见的坑是时间戳或当前状态放在 prompt 开头——每秒都在变，前缀永远匹配不上，缓存命中率归零。

【周迅】那 context 能放多大，是受内存限制吗？

【窦文涛】对，第四个心智模型：Context Window 的限制。KV Cache 要住在 GPU 显存里，因为每次生成新 token 都需要访问它。70B FP16 模型光权重就需要约 140GB——几乎占满一张 H100 141GB 的全部显存，大多数部署必须多卡并行。支持 128K context 的 KV Cache 还需要再加约 64GB（每 1K tokens 占约 0.5GB），这就是为什么大 context 需要大量硬件。

【窦文涛】一个关键工程细节：超过 context window 的内容不会报错——API 会截断最早的内容，静默丢失。agent 在长会话中会突然"忘记"早期的关键指令，原因就在这里。这就是为什么 Claude 200K、Gemini 1M 是工程成就：他们找到了在可控成本下用 KV Cache 压缩和 Multi-Query Attention 等技术扩展 context 的方法。

【周迅】那有没有办法压缩模型，让它住进更小的显存？

【窦文涛】对，第五个心智模型：量化。先说定义——模型权重本质上是一堆浮点数，量化就是把这些浮点数从高精度压到低精度存储，用更少比特表示同一个数值。FP16 是 16 位浮点，每个数占 2 字节，70B 模型光权重就需要约 140GB 显存；INT4 是 4 位整数，每个数只占半字节，70B 模型压到约 35-40GB——这是四分之一的显存。

【窦文涛】量化的作用是什么？消费级 GPU 的入场券。一张 RTX 4090 只有 24GB 显存，四张加起来 96GB，FP16 的 70B 模型根本塞不进去；INT4 量化之后，四张 4090 可以跑 70B 模型。量化的代价是精度损失——INT4 在多数任务质量损失约 5-10%，数学推理受影响较大，分类和摘要基本感知不到。实际例子：你在 Groq 或 Together 上用的那些"便宜大碗"推理服务，底层全是量化模型，这就是它们能比官方 API 便宜三到五倍的原因。

【周迅】量化是压精度，还有没有从架构层面省的办法？

【窦文涛】是的，第六个心智模型：Dense 与 MoE。传统 Dense 模型每次生成 token 时激活所有参数。Mixture of Experts 的思路是：把 FFN 前馈网络层分成多个专家，每次只路由给其中几个处理——如果有 64 个专家每次激活 2 个，实际计算量只有 1/32。但模型"见过的知识"来自所有专家训练——总参数大，激活参数小。DeepSeek-V3 总参数 671B，每 token 激活 37B，2025 年发布时质量接近 GPT-4o，推理成本约为其十分之一。

【窦文涛】关键认知：激活参数决定推理成本，不是总参数。Qwen3-235B 激活约 22B，Mixtral 8×7B 总 46B 激活约 13B——推理成本都远低于总参数量对应的 Dense 模型。当你看到便宜大参数模型，先查激活参数，再做选择。MoE 的代价是所有专家参数都要加载进显存，671B 意味着需要超过 1.3TB 显存存模型——但 API 服务用共享服务器摊平这个成本。

【周迅】了解了底层，落到选哪个模型上有没有实用原则？

【窦文涛】第七个心智模型：选型框架。几条从工具调用实战中沉淀下来的铁律：工具调用首选 Claude Sonnet 系列，function calling 格式最稳定，返回格式遵从率高；中文任务优先 DeepSeek-V3 或 Qwen3，同样内容 Claude 消耗约 1.5 到 2 倍 token，且价格是 Claude 的十分之一以下；本地部署入门：显存 GB 除以 2 约等于 FP16 精度下能跑的参数量 B。

【窦文涛】成本估算在动手之前必须先做：日成本等于每次调用 tokens 乘以调用次数乘以 token 单价。每次调用 8K tokens，每天 1000 次，Claude Sonnet 4.5 输入 3 美元每百万、输出 15 美元每百万，日成本约 36 美元，月约 1080 美元。换 Haiku（输入 0.8、输出 4），月约 288 美元，节省 73%。质量测试先于成本决策——确认 Haiku 满足任务要求再降级。

【周迅】Reasoning 模型跟普通模型选型逻辑一样吗？

【窦文涛】第八个心智模型：Reasoning 模型。先说定义——Reasoning 模型是一类在输出最终答案之前，会先生成大量内部"思考过程"token 的模型，代表有 OpenAI o3、DeepSeek 的推理系列、Claude 的 Extended Thinking 模式。普通模型收到问题直接输出答案；Reasoning 模型会先在内部"想"一大段，然后再给你答案。

【窦文涛】它的作用是提升复杂推理质量——数学证明、多步逻辑、代码调试这类任务，有了思考过程可以反复校验中间步骤，准确率显著高于直接输出。代价是成本和延迟。DeepSeek 的推理模型处理一道多步数学题，内部思考消耗约 2000 到 8000 tokens，然后才输出 50 tokens 的答案——thinking 成本是答案本身的 40 到 160 倍，延迟可能 30 到 60 秒。

【窦文涛】关键限制：Reasoning 模型在 agent 工具调用里格式遵从率不稳定，根本原因在训练目标。普通模型被训练成"需要工具时必须输出这个 JSON 格式"，行为固定。Reasoning 模型被优化成"先把问题想透"，思考过程可以改变主意——思考完后可能忘了要调工具，或者在 thinking tokens 里把答案自己算出来了，然后直接输出文字而不调用工具。你的工具从来没执行，loop 提前终止，用户拿到的是模型内部知识而不是工具实际返回的结果。实战结论：默认用标准模型跑 agent loop，只在"需要深度单次推理"的节点切换 Reasoning 模型——这是专用工具，不是通用升级。

【周迅】想把这些机制真正搞透，有没有推荐的路径？

【窦文涛】Karpathy 的 nanoGPT 是最直接的路——124M 参数 GPT 压缩进 300 行代码，每步都有数字验证，能让你在跑完一遍之后清楚地知道 Attention 计算量从哪里来、KV Cache 缓存了什么。这种"代码跑一遍就懂了"的学习方式，是从听懂到真正建立直觉的关键跨越。下一章，构建"加工具不改核心"的注册机制，让 Lena 优雅扩展到 20 个工具。
