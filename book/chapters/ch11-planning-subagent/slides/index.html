实战框架
四要素 Prompt 框架
写好子 agent 指令的结构化方法
S
Scope 范围
精确告诉子 agent 它的「作战区域」。只查什么，不查什么。
G
Goal 目标
用可验证的结果来表达目标。不要「调研」，要「生成包含 X Y Z 的报告」。
C
Constraints 约束
告诉子 agent「不能做什么」和「怎么做」。工具限制、字数上限、不猜测。
O
Output 输出格式
明确返回格式，方便 Orchestrator 汇总。统一格式 = 汇总不用猜。

设计决策
TodoWrite：纯内存，零 I/O
TodoWriteTool.ts:67  ·  AppState.todos[agentId ?? sessionId]
// TodoWriteTool.ts:67 function writeTodo(agentId, sessionId, todos) { const key = agentId ?? sessionId // 纯内存写入，无任何 I/O AppState.todos[key] = todos // 全完成自动清空 if (todos.every(t => t.status === 'completed')) { delete AppState.todos[key] } }
关键设计
没有 await，没有 fs.writeFile，没有 SQLite。 零 I/O。
草稿 vs 任务系统
维度 TodoWrite（草稿） 持久化任务系统
生命周期 与 session 同生共死 独立于 session
用途 LLM 自己追踪任务 用户/外部系统看
崩溃后 自然清空，重新规划 恢复上次状态
I/O 开销 零 每次写需 I/O
什么时候需要持久化？
任务跨多个 session（"明天继续"）/ 需要外部追踪 / 断点续传（参考 Ch.14）
