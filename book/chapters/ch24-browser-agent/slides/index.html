Browser Agent
为什么是 "通用 agent 压力测试场"？
01
DOM 无穷大
微博首页 ~6000 个 DOM 节点，序列化 800KB。直接塞进 LLM = context 撑满，质量暴跌。
02
页面跳变
SPA 路由切换、表单提交跳转、登录重定向——每次跳转 DOM 完全失效，需要重新感知。
03
反爬 / 人机验证
User-Agent 检测、headless 指纹、行为分析、CAPTCHA——大型平台的多层防御。
04
登录态管理
Cookie 有效期、跨页面 session、多账号切换——agent 需要维护持久的身份上下文。

DOM 感知核心：6000节点 → 50-200可交互元素（体积缩小 30-100x）

可见性过滤：offsetWidth/Height > 0，不处理隐藏元素

6 条血泪教训
1
不能发 Origin 头 → Chrome 403
websockets.connect(ws_url) # 无 extra_headers
2
Clash fake-ip 干扰 socket → 必须清除代理变量
os.environ.pop("http_proxy", None) # 启动时执行
3
并发冲突 → 进程级文件锁
fcntl.flock(fd, LOCK_EX|LOCK_NB) # /tmp/.cdp_scrape_lock
4
tab 不自动关闭 → 200+ tabs 拥堵
async with CDPSession() as cdp: ... # 自动清理 tab
5
空白页截图 <80KB → 跳过而非报错
if len(data) < 80*1024: return None
6
/json/new、/json/close 用 PUT，不是 GET
session.put(f"{CDP_BASE}/json/new?{url}")
