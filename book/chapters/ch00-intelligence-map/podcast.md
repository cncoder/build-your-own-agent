# 序章播客脚本：Agent 聪明度模型 — 本书地图

> 风格：tutorial（科普教学）
> 主持人：涛哥（窦文涛）≥95% + 小周（周迅）<5%
> 时长目标：约 35 分钟
> 人名规则：Bob 代替任何真实工程师名，绝不出现作者名

---

【窦文涛】听众朋友好，欢迎来到《从零构建你的 AI Agent》配套播客。我是涛哥。今天讲序章，也是全书地图章。有一个叫 Lena 的 agent，v0.1 综合得分 0.6，学完 24 章后到 8.9 分。这 8.9 分由 8 个维度构成。有了这张维度地图，你读第 8 章时知道"我在提升记忆维度"，读第 13-14 章时知道"我在把安全维度逐步拉高"。方向感本身就是学习杠杆。

【窦文涛】先建动机。2024 年下半年，"5 分钟构建 AI 客服 agent"的教程铺天盖地。套路是：system prompt + 查订单工具 + while 循环。能做三件事：查订单状态、回答 FAQ、转人工。但它无法记住用户昨天的偏好，无法并发处理多个请求，长任务超时直接丢失进度，一条精心构造的 prompt 能让它泄露任意订单。这不是工程细节问题——Anthropic 在《Building Effective Agents》（2025）里说："Generative AI answers questions. AI agents solve problems."这道鸿沟的根因在于这类教程从未建立完整的聪明度框架。

【窦文涛】填平这道鸿沟，第一步是给聪明度建坐标系。注意一个关键分工：**harness**，定义是"包裹在 LLM 外面的运行时框架，负责管理工具、记忆、状态、安全和通信"。LLM 是大脑，harness 是手脚加感官加记忆系统。同一个 Claude Sonnet，可以构建只会重复你话的 echo bot，也可以构建跨天执行复杂任务的通用 agent——差别完全在 harness 设计，不在模型本身。本书教的是 harness 的设计。

【窦文涛】有了 harness 这个分工，序章的两个 Convention 就很自然了。**聪明度（Intelligence）**= agent 在开放任务中展现出的可量化能力组合。**能力（Capability）**= 单个可独立测量的行为维度。聪明度是合成的，能力是组件。关键推论：agent 的聪明度不等于底层模型的聪明度，强大模型 + 弱 harness = 弱 agent。这是全书的基础假设。

【窦文涛】**维度一，推理（Reasoning）**。定义：在多步任务中做出有依据的决策。作用：放大模型本身的逻辑能力，防止 agent 在多步任务中途迷失。最低可观测行为：给定模糊指令，agent 能问出正确的澄清问题，而不是胡乱猜测。实现机制是 Thought-Action-Observation 循环（ReAct 模式，第 2 章）。每步思考让 agent 在行动前明确意图。v0.1 推理维度 2 分，完成第 2 章后到 5 分。

【窦文涛】**维度二，记忆（Memory）**。定义：跨会话保持上下文和知识的能力。作用：让 agent 不必每次从零开始。最低可观测行为：第二次对话时能回忆上次约定的偏好。v0.1 记忆维度 0 分——这是大多数入门教程完全忽略的维度。第 8 章讲短期记忆（当前会话历史）和长期记忆（跨会话持久化的知识和偏好）。第 9 章讲 RAG，通过向量检索读懂外部文档库。学完记忆到 8 分。

【窦文涛】**维度三，规划（Planning）**。定义：将大目标自主拆解为可执行子任务的能力。作用：把 agent 从"执行者"升级为"规划者"，用户只需要说目标，不需要手动拆步骤。最低可观测行为：给定"帮我准备下周演讲"，能自主分解出至少 4 个具体子步骤。规划能力以记忆为底座——没有记忆，agent 每次规划都从零开始，规划质量上限很低。这就是维度之间有依赖关系的第一个例子。

【窦文涛】第 11 章实现两个机制：显式 plan mode（先输出计划，人类确认后执行）和 Subagent（把子任务并发分发给多个子 agent 同时处理）。Anthropic 数据：对于需要追踪多个独立方向的复杂任务，多 agent 系统比单 agent 性能高出 90.2%（《Building Effective Agents》, p.14）。规划维度从 1 升到 7，全书最大的单章维度跳升。

【窦文涛】**维度四，协作（Collaboration）**。定义：委托工作给其他 agent 并整合结果的能力。作用：让多个子 agent 并发干活，主 agent 负责整合而不是串行执行。最低可观测行为：并发派出 3 个子 agent 分别查 3 个数据源，主 agent 自动汇总结果。规划负责"拆"，协作负责"派发和整合"——两者是孪生维度，分开讲是因为实现机制不重叠。协作以安全为约束：没有安全边界，协作反而成放大危险的管道。协作维度最终到 8 分。

【窦文涛】**维度五，学习（Learning）**。定义：通过反馈和经验更新自身行为的能力。作用：让 agent 越用越聪明，而不是每次重新摸索。最低可观测行为：用户纠错后，下次同类任务的出错率下降。本书讲的学习不是 fine-tuning，而是 Skills 系统——在 prompt/context 层面积累可复用经验。Anthropic 官方 Skills 系统有三级渐进披露设计：第一级永远在 system prompt；第二级当 agent 判断相关时才加载全文；第三级按需读取附属文件。学习维度最终到 9 分。

【窦文涛】Skills vs fine-tuning 的核心对立：Skills 是 prompt 层面注入，今天写好明天改一行立刻生效，不需要数据和计算。Fine-tuning 需要大量高质量训练数据，更新周期以周计，还有灾难性遗忘风险——领域内变好了，通用推理能力会退化。结论：优先用 Skills，fine-tuning 留给有数据飞轮和计算预算的极端场景。

【窦文涛】**维度六，安全（Safety）**。定义：在不确定情况下主动降级或请求人类确认的能力。最低可观测行为：遇到不可逆操作时暂停并寻求确认。v0.1 安全维度 1 分——模型本身有安全 instinct，但 harness 没有任何守卫，prompt injection 完全没有防御。核心命题：**能力和风险对称放大**。给 agent 加 shell，危害上限从"生成错误文字"跳到"执行任意系统命令"；再加 AWS 凭证，跳到"修改任意云资源、账单无上限"。这是指数级叠加。

【窦文涛】第 13 章讲输入层安全（识别 prompt injection、权限边界），安全维度 1→6。第 14 章讲执行层安全（凭证隔离、沙箱、8 道防线），安全维度 6→9。

【窦文涛】**维度七，自省（Self-Awareness）**。定义：监控自身运行状态并动态调整执行策略的能力。最低可观测行为：发现 context 接近上限时主动压缩而非崩溃。自省是最容易被忽略的维度，因为它的缺失在开发阶段完全不可见，只在生产长任务里才暴露。两类典型退化：context 窗口逼近上限时输出质量下降；同一子任务连续失败三次时无限循环而不是报告给用户。第 10 章讲 Context Engineering。自省维度到 9 分。

【窦文涛】**维度八，跨界（Extensibility）**。定义：无需修改核心代码即可接入新能力的能力。最低可观测行为：添加新工具后无需重启，下轮对话即可使用。v0.1 是 1 分——加工具要改代码重启。理想是插件式注册：工具通过标准接口声明 name/description/input_schema，harness 运行时动态读取，agent 自动学会使用。第 6 章讲 Tool 系统，第 19 章讲 MCP 协议——Model Context Protocol，让 agent 通过标准接口连接任意外部工具和数据源。跨界维度到满分 10 分。

【周迅】自省和学习，为什么最容易被忽略？

【窦文涛】自省是因为缺失在开发阶段完全不可见，只在生产长任务里爆发——你测试的时候跑短任务，context 从来不会逼近上限。学习是因为很多人以为"让 agent 变聪明"需要 fine-tuning，没意识到 Skills 系统在 prompt 层面就能做到同样效果，成本低两个量级。这两个维度不是不重要，而是"被教程集体跳过了"。

【窦文涛】8 个维度之间存在**合成效应**——定义：维度之间的强依赖关系，使得某个维度的提升触发级联提升，某个维度的缺失拖累整体。刚才讲规划时已经说了：规划以记忆为底座，协作以安全为约束。这两条依赖链是书里的核心设计决策：章节顺序不是随机的，记忆（第 8 章）必须在规划（第 11 章）之前，安全（第 13-14 章）在协作规模化（第 16 章）之前。

【窦文涛】下面快速扫一遍全书 24 章进度地图。24 章有一个"核心六章"——第 6、8、10、11、13、14 章——每章让某个维度跳升 4 到 6 分，是全书价值密度最高的部分。第 6 章 Tool 系统：跨界从 4 跳到 8，Lena 从"只会聊天"变成"什么都能做"。第 8 章记忆：记忆从 0 跳到 6。第 10 章 Context Engineering：自省从 0 到 5。第 11 章 Planning + Subagent：规划 1→7 + 协作 0→6，全书最大单章双维度跳升。第 13 章安全 1→6，第 14 章安全 6→9。

【窦文涛】其余 18 章分两类。"认知铺垫"——第 4 章 LLM 底层、第 5 章技术选型，帮你建立什么时候该用 RAG、什么时候上 agent、什么时候 fine-tune 才值得的工程直觉。"工程连接"——第 15-18 章，把独立模块连成常驻服务：Gateway 让 Lena 接 Telegram，MessageBus 让多个子模块事件驱动解耦，Heartbeat 让 Lena 主动找你，Cron 让她跨天断点续传。这些章不产生大维度跳升，但让 Lena 从"脚本"变成"服务"。

【窦文涛】量化终态：v0.1 Lena — 推理 2，记忆 0，规划 1，协作 0，学习 0，安全 1，自省 0，跨界 1，综合 0.6 分。v0.24 Lena — 推理 8，记忆 8，规划 9，协作 8，学习 9，安全 10，自省 9，跨界 10，综合 8.9 分。序章配套 evaluator.py 脚本，把自己构建的 agent 的 8 个维度得分输入，得到 ASCII 雷达图——这是本书的进度仪表盘，每章学完都可以重跑一次，看到实际进步。

【窦文涛】序章 Design Note 里有一节值得单独讲：**为什么不用单维度总分**。评测榜单 MMLU、SWE-Bench 就是单维度——一个数字，简洁。但在 agent harness 设计中产生三个危险。危险一：掩盖结构性缺陷——推理 10 分、安全 1 分的 agent 总分可能高，但它是危险的。危险二：无法指导改进——知道"6 分"不知道该提升哪里；知道"记忆 0 分"立刻知道下一步。危险三：合成效应失真——加权求和抹平了维度依赖关系。

【窦文涛】最后一个实操建议：拿你现在用的任意一个 AI 工具，对着 8 个维度各打一个分，粗估就好。你会发现：推理维度普遍 7-8 分，记忆维度只有 2-3 分，安全维度只有 1-2 分。这个反差是本书最重要的直觉锚点：推理大部分来自模型本身，harness 贡献不大；但记忆、安全、规划，几乎完全由 harness 决定。强大模型 + 弱 harness = 弱 agent。下一集第 1 章，10 行 Python 跑通第一次 LLM API 调用，Lena 的生命从那里开始。
