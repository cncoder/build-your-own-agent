# 第 9 章播客脚本：RAG 与向量检索——教 Lena 读懂你的文档

> 时长目标：约 35 分钟
> 人名规则：Bob 代替任何真实工程师名，绝不出现作者名

---

【窦文涛】第九章，我们把 Lena 从 v0.8 升级到 v0.9——这一章她第一次能读懂外部文档。具体说：一个工程师把公司的 200 页 API 规范扔给 Lena，问她"速率限制配置在哪里"，她能找到第 4.3 节那段精确的文本并引用出来。工具定义只有 12 行 Python，基础设施是一个你跑 PostgreSQL 就已经有的 Docker 服务。

【窦文涛】这一章的核心技术叫 RAG——Retrieval-Augmented Generation，检索增强生成。Lewis 等人 2020 年在 arxiv:2005.11401 发表的论文里把这个范式命名并系统化了，思路其实不复杂：与其把整篇文档都放进上下文，不如先检索出最相关的几段，把这几段喂给 LLM，让它基于这些内容回答。先检索，再生成。

【窦文涛】但"直接加个 RAG"是没有意义的建议。RAG 是一条流水线，包含五个独立阶段，每个阶段有自己的设计选择：分块、嵌入、存储、检索、可选重排。每个阶段可以独立地做对或做错——优秀的嵌入模型放在糟糕的分块上结果照样糟糕。你需要对每个阶段单独推理。

【窦文涛】在讲具体实现之前，先建动机。为什么不能直接把 200 页文档塞进 system prompt？有的人会说：现在 Claude 支持 200K tokens，一本书塞得下，为什么还要 RAG？

【窦文涛】这个问题值得用数字回答。一份 200 页的技术文档，中英文混排，大约 15 万 tokens。Anthropic Claude Sonnet 定价，输入 $3/MTok，每次提问光输入成本约 $0.45。一天 1000 次提问，每天 $450，每月 $13500。用 RAG 检索 5 个文本块，输入成本约 $0.02/天。第一堵墙：成本。

【窦文涛】第二堵墙：上下文腐烂（Context Rot）。Anthropic 工程博客《Effective Context Engineering for AI Agents》（2026）里有个关键观察：随着上下文增长，模型对中间段落信息的召回率会下降。一个事实如果藏在 10 万 token 上下文的第 8 万 token 处，召回准确率仅为它在第 5000 token 处时的约 60%。你不会看到崩溃，只有质量悄悄下滑。这是生产环境里最危险的那类 bug——没有报错，只有答案在不知不觉中变差。

【窦文涛】第三堵墙：延迟。15 万 token 的 prompt 首字延迟 3–8 秒。用户发了一个问题，等 5 秒才开始看到响应。RAG 方案的延迟：向量检索 20ms + 5 个文本块拼接 + 模型生成，端到端约 800ms–1.5 秒。

【窦文涛】RAG 存在的意义就是对抗上下文腐烂。与其喂入整篇文档，不如每次提问只检索 3–5 个最相关的文本块，让活跃上下文窗口保持小且信号密集。

【窦文涛】理论铺垫先做三节，然后再看代码。

【窦文涛】第一节：分块（Chunking）。分块做差了，没有哪个嵌入模型能救你。三种策略覆盖了 95% 的使用场景。固定大小（Fixed-size）：每 N tokens 切一刀，重叠 M tokens。优点是实现简单；缺点是可能在句子中间切断，破坏段落结构。语义分块（Semantic）：在句子边界切割，合并直到达到 token 上限。保留意义单元，适合大多数结构化文档。延迟分块（Late Chunking）：嵌入完整文档上下文后再切割所得嵌入，适合有大量交叉引用的技术文档，但需要支持该特性的模型，比如 jina-embeddings-v3。

【窦文涛】有一个数字值得记住：超过约 512 tokens 的文本块会被大多数嵌入模型静默截断。BGE-M3 用正确设置支持最多 8192 tokens，但更小的文本块能给出更好的检索精度——关键句子周围的无关文本更少。本章用语义分块，目标 400 tokens、80 token 重叠，这能处理绝大多数 PDF 和文本文档。

【窦文涛】选择分块策略的决策表：如果文档有清晰结构（标题、章节、编号条款），先按标题切，再在每节内做语义分块，最适合 API 文档和法律合同；如果文档是纯散文，语义分块，256–512 tokens，约 20% 重叠；如果需要逐字引用或页码标注，固定大小，在元数据中记录页码和字符偏移；如果文档是代码，按函数或类的边界切割，而非 token 数量——一个 200 行的函数是一个语义单元，从中间切开会破坏检索。

【窦文涛】第二节：嵌入模型（Embedding Model）。嵌入模型把一段文本转换为稠密向量——一列浮点数。含义相似的文本块最终得到相似的向量，在余弦空间中距离相近。这种映射的质量决定了检索质量的上限。

【窦文涛】三个模型覆盖大多数团队的现实选择。OpenAI text-embedding-3-large：3072 维，$0.13/M tokens，多语言，最大 8191 tokens，适合已在用 OpenAI 且想要托管 API 的场景。

【窦文涛】BAAI/bge-m3：768 维，完全免费，本地运行，100+ 语言，最大 8192 tokens。这是 sentence-transformers 库里使用最广泛的多语言模型之一，在 MTEB 基准上中文检索任务表现稳定。本章的默认选择。Voyage AI voyage-3：1024 维，$0.06/M tokens，最大 32000 tokens，Anthropic 自己的 RAG cookbook 用 Voyage AI 做检索。

【窦文涛】一个容易被低估的属性：相似度分数在不同模型间没有校准。OpenAI 嵌入下的 0.80 分和 BGE-M3 下的 0.80 分质量不同。你的"低置信度"阈值必须针对每个模型通过实验确定，不能套用网上看到的数字。

【窦文涛】Convention 定义：余弦相似度（cosine similarity）= 两个归一化向量的点积，取值范围 -1 到 1。pgvector 的 <=> 运算符计算余弦距离（1 - 相似度）；结果 0.15 意味着相似度 = 0.85。

【窦文涛】第三节：向量数据库。存储向量并高效查询。三个现实选项：pgvector——Docker 或已有 PostgreSQL，5–50ms，不调优可到约 100 万行，已在用 Postgres 就用这个。Chroma——进程内或 Docker 服务，2–20ms，10 万行内表现好，50 万以上开始变慢，适合原型阶段。Qdrant——Docker 或托管云，1–10ms，专为 1000 万+向量设计，规模化生产使用。

【窦文涛】决策树：已经有 PostgreSQL 吗？有，用 pgvector，无新基础设施，ACID 事务，熟悉的 SQL。没有，这是原型还是长期系统？原型用 Chroma，pip install chromadb，零配置，内存模式。长期系统，向量数量可能超过 50 万吗？会超过，用 Qdrant。不会，用 pgvector，从简单开始，必要时再迁移。本章用 pgvector，一个 Docker 容器，标准 SQL，无新服务。

【窦文涛】现在讲一个大多数 RAG 教程都会错过的技术。它来自 Anthropic 的 Contextual Retrieval（情境检索）博客文章（2024 年 9 月）。核心洞见：当你孤立地嵌入一个文本块时，向量只捕获文本块内部的内容，而不是该文本块在整篇文档语境中的含义。

【窦文涛】举个具体例子。一个文本块写着"参数设置为 max_retries=3"。这段文字在语义上是模糊的——它可能来自任何文档的任何章节。当你查询"针对 /v2/completions 端点的重试配置"时，查询向量和该文本块向量之间的余弦相似度低于应有水平，因为两个向量都缺少上下文信息"这是关于 /v2/completions 端点的"。

【窦文涛】解法是在嵌入前为每个文本块附加一段简短的定位句。嵌入前：只有"参数设置为 max_retries=3"。加入情境检索后变成："[来自 API 规范的 /v2/completions 端点配置章节] 参数设置为 max_retries=3"。两段文字嵌入得到的向量质量差别非常大。

【窦文涛】Anthropic 在九个代码库上的量化评估结果：top-20 文本块检索失败率降低 35%，Pass@20 从约 87% 提升到约 95%。成本问题通过 prompt 缓存解决：把完整文档标记为 cache_control: ephemeral，第一个文本块时文档写入 KV 缓存（小额溢价），同一文档的每个后续文本块从缓存读取，享受 90% 折扣。对于一个 10000 token 的文档包含 25 个文本块，不使用缓存总成本约 $0.063，使用 prompt 缓存约 $0.014——便宜 78%。这项技术不是 LangChain 抽象，它是一个 20 行代码就能实现的 prompt 模式。

【窦文涛】理论讲完，进代码。先搭骨架，后面逐步填充。

【窦文涛】数据库表结构（schema.sql）：扩展 pgvector，建 chunks 表，字段有 id、doc_id（文档标识）、chunk_index（在文档中的位置）、content（原始文本块）、context（情境检索前缀，可为空）、embedding（VECTOR(768)，BGE-M3 输出）、metadata（JSONB）。再建 IVFFlat 索引，lists=100，这能处理约 10 万行规模的检索。对于更大数据集，lists 设为 sqrt(行数)。

【窦文涛】Python 摄入骨架（ingest.py）：用 psycopg 连接数据库，ingest_document 函数接收 doc_id 和 text，Beat 4 阶段只打印占位符，先验证导入链干净，再逐步添加真正的逻辑。增加复杂性之前先验证骨架能运行，这是防止"改了半天才发现环境问题"的标准工程实践。

【窦文涛】每次只添加一个能力，每步之后代码都能运行并打印有意义的内容。扩展一：分块。核心函数 chunk_text：用 NLTK 做句子分割，贪心合并句子直到接近 400 tokens，保留 80 token 重叠。Token 计数用 1 token ≈ 4 字符的粗略估算——对于决定切割点已经够用，不需要精确的 tokenizer。

【窦文涛】扩展二：情境生成。函数 generate_context 接收完整文档内容和单个文本块内容，调用 claude-haiku-4-5（生成情境快且便宜），使用 cache_control: ephemeral 标记文档块，为该文本块生成 1–2 句定位描述。确认缓存生效的方法：第一个文本块之后，response.usage.cache_read_input_tokens 会跳升到文档长度。

【窦文涛】扩展三：嵌入加 pgvector 插入。embed 函数用 sentence-transformers 的 SentenceTransformer("BAAI/bge-m3")，normalize_embeddings=True，返回 768 维向量列表。insert_chunks 批量插入，对每个文本块拼接 context 前缀和 content 再嵌入，用 psycopg 的 executemany 批量写入——逐条 INSERT 比批量慢约 10 倍。FAISS 这类纯内存向量库在这里不适合，因为需要持久化存储和持续更新文档。

【窦文涛】扩展四：检索策略。三种主导策略：Top-K 余弦——返回 K 个最近向量，快速简单，即使结果冗余也会返回；MMR（最大边际相关度）——平衡相关性与多样性，惩罚与已选文本块相似的候选项，减少冗余但约 2 倍复杂度；混合检索（BM25 + 向量）——通过 RRF（倒数排名融合）组合关键词匹配分和余弦分，能捕获精确关键词匹配，需要额外基础设施。

【窦文涛】选择决策：如果文档是密集散文，Top-K 足够；如果文档包含精确技术术语、模型名称、错误码（比如 claude-opus-4-7 或 RATE_LIMIT_EXCEEDED），加混合检索；如果 Top-K 结果高度冗余，考虑 MMR。本章主流水线用 Top-K 余弦相似度，对结构化文档已足够，不需要额外基础设施。

【窦文涛】Anthropic 的情境检索评估显示，加入 BM25 后代码库查询的 Pass@20 从约 95% 提升到约 95.2%——对于大多数纯文本文档提升有限，但对包含精确标识符（API 路径、参数名、错误码）的文档提升明显。

【窦文涛】search_knowledge_base 函数：接收 query（自然语言查询）、top_k（默认 5，超过 10 通常只添加噪音）、doc_id（可选，过滤到特定文档）。实现：query 通过 BGE-M3 编码成查询向量，pgvector 的 <=> 运算符做余弦距离排序，LIMIT top_k。关键安全阀：检查所有返回文本块的最高相似度，如果 < 0.50 加上警告"检索到的文本块可能不相关，请考虑换个方式提问"——检索失败不应该静默。

【窦文涛】扩展五：最小可行评测。这节故意写得简短，第 21 章会覆盖完整评估流水线，但现在需要建立最小信号：上线前知道系统是否在工作。Convention 定义：Recall@K = 相关文本块出现在 top-K 检索结果中的比例；Precision@K = top-K 结果中相关的比例。好的检索系统优化 Recall@K——LLM 可以过滤噪音，但无法凭空创造没有检索到的信息。

【窦文涛】最小可行评测：20–50 道带有已知正确文本块的测试题。构建方法：摄入一篇文档，手工识别哪个文本块回答了哪道测试题。Recall@5 低于 70%，问题几乎总是在分块上（文本块太大、语义边界被破坏）或文档质量上（OCR 错误、无法提取文本）。在这个规模下，嵌入模型和检索策略很少是瓶颈。

【窦文涛】跑起来看看。docker-compose.yml 只有一个服务：pgvector/pgvector:pg16 镜像，环境变量设好用户名、密码、数据库，挂载 schema.sql 到 docker-entrypoint-initdb.d。执行顺序：docker compose up -d 启动，pip install 依赖，运行摄入脚本，运行查询脚本。

【窦文涛】完整摄入流程的预期输出：读取 12 页、提取 8421 字符、切分成 22 个文本块（平均 383 tokens）、生成情境（第一个文本块写入缓存，后续文本块全部走缓存读取，90% 折扣生效）、BGE-M3 嵌入 22 个文本块、批量写入 pgvector 耗时 0.31 秒，总时间约 18 秒。

【窦文涛】查询示例输出格式：[sample §4 | similarity=0.847]，上面是情境前缀，下面是原始文本块内容。相似度 0.847 是一个健康的分数，说明检索命中了正确段落。

【窦文涛】常见失败模式逐一过一遍。connection refused——docker compose 还没启动，等 5 秒再运行。"type vector does not exist"——schema.sql 没有执行，手动运行 psql 初始化。SentenceTransformer 找不到模型——第一次运行下载 BGE-M3 约 560MB，等网络下载。相似度分数全在 0.5 以下——通常意味着文档没有相关内容，或文本块太大（超过 BGE-M3 默认的 512 token 截断限制；换用带 max_length=8192 的配置或用更小的文本块）。

【窦文涛】把 search_knowledge_base 注册成 Lena 的第五个工具。工具定义：name、description 说明何时用（需要回答关于已摄入文档的问题时）、input_schema 三个参数（query 必填、top_k 默认 5、doc_id 可选）。注册后 Lena 能处理："根据我们摄入的 API 规范，metadata 字段的格式是什么？"——她会调用 search_knowledge_base，拿回相关文本块，在不把任何文档放进 system prompt 的情况下综合出答案。

【周迅】涛哥，为什么不直接把文档塞进上下文窗口就好了，感觉 RAG 这套挺复杂的？

【窦文涛】这个问题我在 Beat 2 里给了三个数字：成本、上下文腐烂、延迟。真正的分水岭是规模：如果文档不超过 20 页，用户每天只问几个问题，直接塞进上下文窗口是正确的选择，更简单，没有基础设施开销。RAG 的成本优势在高频率、大文档的场景下才真正显现——每天 1000 次提问对 200 页文档，RAG 方案每月省约 $13480。复杂度是负债，需要有收益覆盖它才值得引入。

【窦文涛】两个 Design Note 值得在收尾时单独说一遍。

【窦文涛】Design Note 1：为什么不直接塞满上下文窗口。成本和上下文腐烂已经讨论了。再补充一个梯度曲线：这不是硬性限制，而是一条渐降曲线。10 万 token 上下文里第 8 万 token 处的事实，召回准确率比第 3000 token 处低约 40%。你不会看到崩溃，只有质量慢慢下滑。

【窦文涛】建议：文档不超过 20 页、每天只问几个问题，直接塞进上下文窗口更简单。对于更大规模或更高流量，RAG 的成本和质量优势是明确的。生产部署时还要加监控（追踪相似度分数分布；分数持续偏低说明查询出了分布外）、嵌入缓存（对重复查询缓存向量）、降级方案（检索置信度低时回退到全文档填充）。

【窦文涛】Design Note 2：为什么 Agent + RAG 是 LLM 应用的默认形态。RAG 单独不够用——它只能回答"文档说了什么"，无法决定要搜索哪个文档、何时搜索 vs 使用自身知识，或者检索到的答案是否可信到足以采取行动。

【窦文涛】Agent 外壳补上了缺失的一层：用户问题进来，Agent 决定是知识库里的还是常识，如果是 KB 则调用 search_knowledge_base，评估文本块相关性，置信度低时要求澄清或回退到训练知识，最后综合答案并引用来源。

【窦文涛】Anthropic 在 Building Effective Agents（2024-12-19）里说得直接："从最简单的解决方案开始，只在必要时才增加复杂性。"本章的 pgvector 加单模型方案，无需修改可以处理最多约 10000 页的文档。在你有证据证明简单版本不够用之前，不要添加专用重排器、向量数据库集群或 BM25 混合层。

【窦文涛】附录里还有重排器的内容。重排器是可选的第二阶段打分。决策规则：top-5 相似度全在 0.75 以上，跳过，检索已经高置信度；聚集在 0.55 到 0.70，加重排器，它把真正相关的提升上来；相似度低于 0.50，重排器救不了，问题在检索本身。

【窦文涛】两个重排器选择：Cohere rerank-english-v3.0，$2/1K 次查询，约 100ms，适合纯英文；BAAI/bge-reranker-v2-m3，免费本地，CPU 约 200ms，多语言。大多数文档不需要重排器——加之前先衡量它是否真的提升了端到端答案质量。

【窦文涛】章节小结。Lena v0.9 现在能读一本 200 页的 PDF 并回答关于它的问题了。六步流水线：语义分块（400 tokens，80 token 重叠）→ 情境检索生成上下文前缀（失败率降 35%，prompt 缓存近乎免费）→ BGE-M3 嵌入（768 维，免费）→ 存入 pgvector。

【窦文涛】余弦相似度检索返回 top-K，可选重排。search_knowledge_base 工具 40 行代码，基础设施是你跑 PostgreSQL 就已经有的 Docker 服务。这不是框架——是三条 SQL 查询加一次嵌入调用。
