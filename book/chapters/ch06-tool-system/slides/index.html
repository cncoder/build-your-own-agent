read_file · 只读 · 并发 · ∞
@registry.tool(description="读文件", is_read_only=True, is_concurrency_safe=True, max_result_size_chars=None) # ∞ def read_file(input: ReadFileInput) -> str: path = Path(input.path) if not path.exists(): return f"错误：文件不存在" content = path.read_text(encoding=input.encoding) lines = content.splitlines() return f"=== {input.path} ({len(lines)} 行) ===\n{content}"
write_file · 破坏性 · 串行
@registry.tool(description="写文件", is_read_only=False, is_destructive=True, # Ch10 会拦截 is_concurrency_safe=False, # 串行防竞态 max_result_size_chars=500) def write_file(input: WriteFileInput) -> str: path = Path(input.path) path.parent.mkdir(parents=True, exist_ok=True) mode = "a" if input.append else "w" with open(path, mode, encoding="utf-8") as f: f.write(input.content) return f"成功写入 {len(input.content)} 字符"
shell · 白名单安全 · 破坏性 · 串行
BLOCKED = ("rm ", "sudo ", "| bash", ...) ALLOWED = ("ls", "cat", "grep", "find", ...) @registry.tool(description="执行 shell（白名单）", is_destructive=True, is_concurrency_safe=False) def shell(input: ShellInput) -> str: for blocked in BLOCKED: if blocked in input.command: return "🚫 安全阻断" if not any(input.command.startswith(p) for p in ALLOWED): return "⚠️ 不在白名单" result = subprocess.run(input.command, ...) return result.stdout
web_search · 只读 · 并发 · DuckDuckGo
# DuckDuckGo Instant API，无需 Key @registry.tool(description="搜索互联网", is_read_only=True, is_concurrency_safe=True, # 并发！ max_result_size_chars=3000) def web_search(input: WebSearchInput) -> str: url = (f"https://api.duckduckgo.com/" f"?q={urllib.parse.quote(input.query)}&format=json") with urllib.request.urlopen(url) as resp: data = json.loads(resp.read()) return format_results(data)

Ch 04 · Tool 系统 8 / 11
案例 4.2
类型 Bug 的教训
HA 华为空调 MQTT Bug
控制器 A 发送：
# 控制器 A（Python bool） {"on": true} # 解析器 B 期待 {"on": 1} # 或 {"on": "true"}
✕ 空调不响应，没有报错，没有日志
根因：类型不一致。
true（bool）vs 1（int）在 JSON 是不同值，但人类看都是"开"。
Pydantic 如何预防这类 bug
class ACControlInput(BaseModel): on: bool = Field(description="开机：True/False") temperature: float = Field(ge=16, le=30) # LLM 传入不规范的值 validated = ACControlInput(**{ "on": 1, # int "temperature": "24" # str }) # Pydantic v2 自动转换 print(validated.on) # True (int→bool) print(validated.temperature) # 24.0 (str→float) # 需要严格模式？ class StrictInput(BaseModel): model_config = ConfigDict(strict=True) on: bool # ACControlInput(on=1) → ValidationError!
教训：Pydantic 验证层是工具输入的
最后一道防线——把运行时 bug
变成清晰的 ValidationError
Ch 04 · Tool 系统 9 / 11
Design Note
为什么不用函数直接 eval？
🔒
安全
直接 eval LLM 生成的代码：

LLM 幻觉 → 执行任意代码

工具系统：白名单注册，
只有已知工具可执行，
输入经 Pydantic 验证。

参考 R2 nanoClaw ShellSandbox
三层过滤（BLOCKED/CONFIRM/EXECUTE）
# ❌ 危险 eval(llm_generated_code) # ✓ 安全 registry.execute("read_file", inputs)
🎛
权限
直接 eval：
无法拦截、无法审批、
无法回溯。

工具系统：
五步决策流程 在执行前
给用户确认机会。

三标志让每个工具的
安全意图在声明时就确定。
isDestructive=True
→ Ch10 弹框确认
→ 审计日志记录
→ 可撤销机制
⚡
并发
直接 eval：
并发时难以控制执行顺序，
竞态条件无法预防。

工具系统：
isConcurrencySafe 标志
让调度器精确控制——
只有安全的工具才并发。
