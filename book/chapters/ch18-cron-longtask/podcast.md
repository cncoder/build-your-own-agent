【窦文涛】今天讲一个听起来平凡但藏着大坑的话题——长任务崩了怎么办。我说的不是崩溃后重试那一两秒的任务，我说的是跑了十一个小时突然被 kill -9 杀死的 Agent 进程，那十一小时的工作全进了垃圾桶，它只能从第零步重来。这是 Lena 从 v0.17 升到 v0.18 要解决的核心问题——怎么让一个跨天运行的新闻流水线在任何时刻被杀掉之后，都能从断点精确恢复，一条数据不多跑，一条数据不少做。

【窦文涛】先说 Lena 此刻在 v0.17 能做什么。她有网关、有 MessageBus、有 Heartbeat，可以每天早上打个招呼——这是一次性动作，一秒之内就完成，失败了明天重试，什么都不会丢。本章要处理完全不同的一类任务：一个 24 步流水线，每小时抓取一次新闻，持续整整一天，积累 24 批次，午夜汇总所有内容。如果进程在第 17 步崩溃，正确行为是从第 18 步恢复——不是从第 1 步重来。这一章教会读者把这种韧性长在自己 agent 上。

【窦文涛】本章脉络有两条主线。第一条是外部状态：croniter 无漂移调度加 SQLite 断点引擎，确保进程在任何时刻被杀后都能从最后成功的步骤恢复。第二条是内部状态：LLM 自己的 context window 在长任务里会膨胀到爆，光有外部断点不够，还要搞清楚 Compaction 和 Context Reset 哪个才是正确的应对。两条主线最后汇合——一个真正能跨天运行的 agent 需要外部断点加内部 context 管理两层同时到位。

【窦文涛】动机从一段坏代码开始。设想你午夜启动一个简单循环：遍历 24 次，每次调一次 fetch_news 等约两秒，再 sleep 3600。上午十一点——已经跑了十一小时、拿到了十一批次——服务器打内核补丁重启。articles 列表在内存里，进程一死，列表清零，流水线从第零小时重来。浪费的不只是十一次 API 调用，还有十一个小时，摘要晚了一整天。如果是付费新闻 API，每次调用花 $0.005，这一次崩溃直接浪费 $0.055 并延误报告 24 小时；换成每步 $0.50 的 LLM 密集型流水线，单次崩溃就是 $5 的真实损失。

【周迅】就算重跑，数据还能补回来，这损失听起来可以接受——时间损失之外还有更深的问题吗？

【窦文涛】有，而且更隐蔽——时钟漂移。如果 fetch_news 耗时四秒，流水线在午夜、凌晨 1:00:04、凌晨 2:00:08 运行。到中午，调度已经比预定的每小时节奏漂移了 1.6 分钟。进程重启时没有"上次什么时候运行"的记录，也就没法知道宕机期间有没有错过该跑的那一次。croniter 解决这两个问题——以绝对墙钟时间定义计划，提供 get_prev() 方法回答"我宕机期间本应运行什么"。

【窦文涛】cron 表达式是五个空格分隔的字段：分钟、小时、月中第几天、月份、星期几，全部通配符用星号。0 * * * * 表示每小时第 0 分触发，0 0 * * * 表示每天午夜触发，\*/15 * * * * 表示每 15 分钟触发。croniter 是 Python 版解析器，不替代系统 crontab，而是让调度逻辑跟着代码走——可以版本控制，可以在常驻进程里维护状态，这是把调度控制权从操作系统层拉回应用层的关键一步。

【窦文涛】调度器的骨架紧凑到 40 行。Scheduler 类持有一个 jobs 列表，每个 job 记录 cron 表达式、触发函数、下次运行时间和 croniter 迭代器对象。run_forever 每秒检查 now >= job.next，满足则触发，捕获异常防止一个任务失败影响其他任务，然后无论成败都调 get_next 推进下次时间。关键点：构造器里 croniter(expr, datetime.now()) 以当前时刻为锚点，第一次 get_next() 给的是锚点之后的下一个未来触发时间——在 12:47 启动、表达式 0 * * * * 的调度器，第一次抓取在 13:00，不在 12:00。这是预期行为。

【窦文涛】现在比较三种调度原语，因为这个位置插一张对照表最值得记住。time.sleep(N) 是最简单的：等固定间隔再运行。问题是漂移——任务本身耗时会累积偏差——以及失忆——重启后不知道上次跑没跑。第十七章 Lena 的 Heartbeat 使用 setTimeout 风格回调，比 sleep 稍好，但仍然没有固定墙钟计划概念，崩溃后同样不知道错过了什么。croniter 加墙钟轮询：无漂移，有 get_prev() 感知错过触发。结论——简单轮询和重试用 sleep，主动通知用 Heartbeat，要按日历精确触发并且能检测漏跑就用 croniter。

【周迅】调度解决了"什么时候跑"，但"跑了一半崩掉"还是要重头来——怎么记住跑到哪了？

【窦文涛】断点引擎要满足三个不变式，一个都不能少。不变式一：幂等性——重新运行一个已完成的步骤必须安全，不产生副作用。实现方式：在 task_id 加 step_id 上建复合 PRIMARY KEY，insert 语句写成 INSERT OR IGNORE。INSERT OR IGNORE 的含义是"如果这一行的主键已经存在，整条语句静默跳过，不更新也不报错"。函数返回 cursor.rowcount：等于 1 说明新插入，等于 0 说明已存在，调用者检查这个值来决定是否跳过后续逻辑。这就是"最多写一次"的 SQL 惯用写法。

【窦文涛】不变式二：原子性。步骤结果和"步骤已完成"标记必须在同一个事务里一起落盘。如果你先写结果再写标记，进程在两次写之间崩溃，重启后步骤再次运行，可能产生重复副作用——多发了一条 Slack 通知，多插了一行数据库记录，多触发了一次扣款。SQLite 的事务语义免费给你这个保障。不变式三：确定性步骤标识。每个步骤需要一个在重新运行时保持不变的稳定标识符。

【窦文涛】为什么选 SQLite 而不是 JSON 文件，不是 Redis？JSON 文件写一半崩溃会给你一个截断文件，下次无法解析，代码可能静默从头开始而不报错。你可以用"写临时文件再 os.replace 原子重命名"规避，但那时候你在重新实现 SQLite 已有功能的子集，而且没有可查询性。

【窦文涛】Redis 快、原子操作、内置过期，但需要运行中的外部服务——断点存储可能在进程启动时不可用，造成复合失败。SQLite 是本地磁盘一个文件，只在磁盘故障时失败，与进程本身失败模式相同。零配置、原生原子、Python 标准库自带。WAL 是 Write-Ahead Log——写操作先追加到日志文件而非直接覆盖数据页，写入者活跃时读取者仍可并发访问旧版本，不互相阻塞。单进程流水线不需要它，但日后若加了独立监控进程需要同时查断点表，加一行 PRAGMA journal_mode=WAL 即可。

【窦文涛】断点引擎代码约 55 行，三个函数。init_db 创建表，CREATE TABLE IF NOT EXISTS 使它完全幂等，每次重启可以安全调用。save_step 持久化一个已完成步骤，INSERT OR IGNORE，返回 True 表示新插入，False 表示已存在。completed_steps 查询某个 task_id 已持久化的所有 step_id，返回 Python set 以实现 O(1) 成员检测。表结构四列：task_id、step_id、result_data（JSON blob）、completed_at，PRIMARY KEY 是 (task_id, step_id) 复合主键。

【窦文涛】验证幂等性用三行交互式测试。init_db 建表。第一次 save_step 返回 True，新插入。第二次用不同数据调同一个 step_id，返回 False，数据未改变。completed_steps 返回 {'step-0'}。这是崩溃恢复所需的全部保障，其余的都从这里推出来。Python 的 sqlite3.connect(path) 作为上下文管理器使用时，正常退出提交，异常时回滚，这与关闭连接不同——如果长时间持有连接，要显式调用 conn.close()。对于本章的短生命周期操作，上下文管理器模式是正确的。

【窦文涛】把两个组件组合进抓取任务后，代码里出现了一个关键的可观测特征：第一次运行打出三行 checkpoint 日志，崩溃重启后第二次运行打出一行 skip 日志加两行 checkpoint 日志——第二次比第一次少一次真实网络调用。这个可见的行为差异是唯一可信的验证手段，比任何单元测试都更贴近生产现实，因为它在真实的崩溃-重启流程里验证了不变式。quick_test.py 提供的就是这个验证入口，不用等一整天。

【窦文涛】内容哈希 step_id 是对基于位置 step_id 的升级。基于位置的 fetch_hn 在任务结构固定时有效。但考虑一个逐篇处理文章正文的流水线——用 LLM 生成摘要。不同日期可能在 Hacker News 顶部返回同一篇文章。用位置 ID，第二天重新处理了数据库里已有结果的内容；用内容哈希 ID，第二天立刻找到现有结果并跳过 LLM 调用。实现是 hashlib.sha256(f"{source}:{article_id}".encode()).hexdigest()[:16]。

【窦文涛】顺序 ID 和内容哈希 ID 可以在同一张断点表共存，因为它们只是字符串。实际规则很直观：如果步骤是"处理第 N 个数据源"，用顺序 ID；如果步骤是"处理这篇特定文章"，用内容哈希 ID。两者选择的依据是：你的步骤是由它的**位置**定义的，还是由它处理的**内容**定义的。

【窦文涛】有一个重要限制：内容哈希缓存假设步骤的输入决定其输出。如果步骤结果依赖于外部状态——今天的汇率、当前天气——昨天的缓存结果就过期了。对于这类步骤，加入 TTL：在结果旁边存储 expires_at 列，在 completed_steps 中将过期行视为不存在。

【窦文涛】午夜汇总任务演示了 SQL 查询的自然用途。all_steps(fetch_task_id) 从数据库按顺序返回该 task_id 所有行，汇总任务聚合今天所有每小时 task_id 下保存的步骤，把文章标题传给 Claude，对结果做断点。开始时的幂等性检查确保 LLM 调用最多发生一次，即使进程在调用之后、断点写入之前重启。这种聚合操作是 SQL 的强项，普通文件或内存字典无法同样干净地支持。

【窦文涛】模块分离带来的实际好处：core/checkpoint.py 对新闻和调度一无所知，你可以不跑任何 cron 任务，只用 quick_test.py 验证 kill -9 后的恢复——这在真实 24 小时流水线里是不可能等一整天去验证的，分离让每个模块可以独立测试。

【窦文涛】说到这里有个微妙的问题。checkpoint 管的是**进程外部**的状态——磁盘有记录，从哪步继续有据可查。但 LLM 还有一个**内部**状态：context window。24 步流水线跑到第 17 步，前 16 步对话历史已经塞满上下文——即使每步都有断点，context 满了 LLM 照样开始犯"健忘错误"，性能悄悄下降。磁盘断点救不了这个。Anthropic 2026 年 3 月做了 harness design 实验，对比了两种策略——Compaction 把旧对话压缩成摘要让 agent 继续跑，保持连续性，但 context anxiety 未解，agent 仍感知"快满了"。

【窦文涛】Context Reset 清空 context，启动新 agent，用结构化 artifact 交接状态，彻底消除 context anxiety，代价是需要设计 handoff artifact。实验结论是：仅靠 compaction 不足以维持长任务性能，必须用 context reset——context anxiety 是真实的，不是理论上的。这意味着真正生产级的长任务 agent 需要三层联动：磁盘断点 + compaction 延缓 context 膨胀 + context reset 在阈值触发时清空并用 checkpoint artifact 交接。第十章、第十七章和本章是一个整体。

【窦文涛】现在做一个重要决策分析。为什么长任务不设计成从零重启？显然的替代方案是让每个任务无状态，进程崩溃时删除所有部分结果从头开始。两种情况下这是合理策略：当每个步骤足够便宜、重新运行一切的成本低于构建断点系统的成本；以及当每个步骤真正无副作用时。

【窦文涛】当任务具备以下三种属性之任意一种，这个权衡就变得不利。第一，昂贵的步骤——一个花 2 美分耗时八秒的 LLM 调用，24 步流水线崩溃在第 23 步，零重启策略浪费 $0.46 和 3 分钟；有了断点，重启只花最后一步的 $0.02。第二，有副作用的步骤——如果步骤发送了 Slack 通知、写入外部数据库、触发了支付，重新运行不安全，"最多运行一次"语义必须靠断点保证。第三，步骤很多的长流水线——一个有 24 步、每步崩溃概率 1% 的流水线，至少一步崩溃的概率超过 21%；有了断点，每次崩溃最多浪费 1 步；没有断点，平均浪费 12 步。

【周迅】这套 SQLite 方案适合单机 agent——流水线变复杂了怎么办，什么时候才需要换工具？

【窦文涛】什么时候升级离开本章的手写方案？有三个明确触发点。第一，任务图出现分支逻辑或并行扇出——本章断点表假设步骤线性有序；第二，需要人在环路中暂停——人工审批后才继续，纯 SQLite 轮询做不到；第三，需要可视化调试界面查看每步状态。出现其中一个，就考虑 LangGraph：它实现相同原则——每节点后写状态，从最后成功节点续传——支持 SQLite、Redis 和 PostgreSQL 后端，权衡是增加约 30MB 依赖。

【窦文涛】再往上，跨机器分布式场景才需要 Temporal，它需要独立服务端，部署成本比 SQLite 高两个数量级。决策树：单机单 agent 每天跑一两次 → 本章 50 行代码够用；出现上述三个触发点之一 → 迁移 LangGraph；多机多 agent → 考虑 Temporal。

【窦文涛】生产版本为什么有 30+ 个模块而不是 lena-v0.18 的五个文件？每一个额外模块背后都是一次真实故障。以最典型的两个为例：cron 任务如果与交互式 agent 共享同一个会话上下文，一次漫长的午夜汇总就能填满 context 窗口，用户第二天早上打开对话发现 agent 失忆——因此需要"会话隔离"模块把 cron 任务的 context 和交互 context 分开。

【窦文涛】另一个：多租户部署里出过一个 bug——为 Agent A 配置的 cron 任务在 Agent B 的 API key 下运行，账单错了人。这需要"认证配置隔离"模块。设计原则的核心——持久化每个步骤，重启时检查——在两个版本里完全相同，三个不变式不变。lena-v0.18 是个人单用户版，已经完整；需要多用户多模型时再叠加。

【窦文涛】给读者一个有新信息的行动建议：**先只装断点，不改其他任何东西**。把 core/checkpoint.py 的 55 行抄进你的项目，在你目前最贵的那个步骤前后加两行——completed_steps 检查加 save_step 调用。然后故意在这两行之间 kill -9 进程，验证重启后是否跳过了这个步骤。这个实验把你从"相信断点有用"变成"亲眼看到断点工作"，是从理解原理到建立工程直觉的必经一步。下一章是 MCP，Lena 将学会在运行时扩展自己的工具集，无需修改代码就能连接任何外部工具服务器。
